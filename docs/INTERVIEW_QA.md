## 项目背景

个人开发的知识库智能问答系统 KB Assistant V3，基于 RAG（Retrieval-Augmented Generation）架构，整合 Dense 向量检索、BM25 关键词检索、知识图谱检索、CLIP 多模态图像检索四路召回，支持文本与图片统一入库、SSE 流式回答、多轮对话查询改写、调试信息面板，并集成了基于 RAGAS 框架的完整性能评估系统。技术栈为 FastAPI + Qdrant + Neo4j + LangChain + 智谱AI GLM-4。

## 第一模块：多路检索与融合策略

### 问题1：四路召回分别解决什么核心问题？

**面试官提问**：在多模态RAG项目中采用了 Dense、BM25、Graph、Image 四路召回，能分别说明它们解决什么核心问题吗？

**回答**：

四路召回分别从不同维度弥补单一检索方式的缺陷，实现互补式检索。具体而言：

- **Dense 向量检索**：基于智谱 embedding-3 API 生成 1024 维文本向量，通过 Qdrant 的余弦相似度搜索完成检索。解决的是深层语义匹配问题——当用户查询与文档存在同义表述或语义关联但字面不同时（如"个人知识库"与"私有文档管理系统"），Dense 能捕捉这种上下文语义关系，克服关键词字面匹配的局限。代码中 Dense 召回 `top_k * 3` 条候选，保证召回覆盖面。

- **BM25 稀疏检索**：基于 `rank_bm25` 的 BM25Okapi 算法，对文本子块做空格分词后构建倒排索引，持久化为 `bm25_index.json`。解决的是精确关键词匹配问题——对实体名、术语、缩写等明确词汇（如"Neo4j"、"CLIP"），BM25 能做到高效精准匹配，弥补向量检索在字面精度上的短板。

- **Graph 图谱检索**：基于 Neo4j 构建 Doc → Entity → RELATED 三层图结构。使用 jieba 的 `extract_tags` 从父块文本中提取 Top-K 实体关键词（过滤长度≥2），建立文档到实体的 MENTIONS 边，以及实体之间的共现 RELATED 边（累计权重）。检索时从用户查询中提取实体，在 Neo4j 中通过实体关联查找文档并累计得分。解决的是多跳逻辑推理和实体关系类查询——当问题涉及实体间的关联关系时（如"A产品和B产品的区别"），图谱能提供结构化的推理证据。

- **Image CLIP 多模态检索**：使用 CLIP ViT-B-32 多语言模型（通过 SentenceTransformer 本地加载）对图片编码为 512 维向量，与文本向量统一存储在 Qdrant 中。支持两种模式：用户上传图片时通过 Image Encoder 编码实现以图搜图；CLIP 模式下通过 Text Encoder 编码查询实现以文搜图。解决的是跨模态检索需求——打破文本单一模态限制，实现图文双向检索。

### 问题2：Graph 召回的实体和关系是如何构建的？

**面试官提问**：Graph 召回的实体和关系是如何构建的？是否依赖特定的结构化数据？

**回答**：

不依赖特定的结构化数据，完全从非结构化文本中自动构建。具体实现在 `core/retrieval/graph_retriever.py` 的 `LightRAGGraphEngine` 中：

1. **实体提取**：对父块文本调用 `jieba.analyse.extract_tags(text, topK=8)`，提取 Top-8 关键词作为实体，过滤掉单字符（`len(w) >= 2`），确保提取的是有意义的实体词。这是一种轻量级方案，无需训练 NER 模型，降低了开发复杂度。

2. **文档节点**：每个父块通过 `MERGE (d:Doc {id:$pid}) SET d.content = $content` 写入 Neo4j，父块文本截断到 4000 字符存储。

3. **Doc-Entity 关系**：通过 `MERGE (n:Entity {name:$name}) WITH n MATCH (d:Doc {id:$pid}) MERGE (d)-[:MENTIONS]->(n)` 建立文档到实体的关联边。MERGE 操作保证幂等性，重复构建不会产生重复边。

4. **Entity-Entity 共现关系**：对同一文档内提取的所有实体做两两配对，通过 `MERGE (a)-[r:RELATED]->(b) ON CREATE SET r.weight=1 ON MATCH SET r.weight=r.weight+1` 建立共现边，权重表示共现次数。共现次数高的实体对，关联性更强。

5. **图谱检索**：从用户查询中提取 Top-5 实体，在 Neo4j 中查找每个实体关联的文档，按命中实体数累计得分，返回 Top-K 文档。同时返回 evidence 列表（entity-doc_id 对），供调试和评估使用。

### 问题3：RRF 的核心公式是什么？为什么项目采用双层融合而非纯 RRF？

**面试官提问**：能解释一下 RRF 的核心公式，以及为什么它比简单的加权求和更适合多路召回场景吗？

**回答**：

RRF（Reciprocal Rank Fusion）核心公式为：

$$Score(d) = \sum_{i=1}^{n}\frac{1}{k + rank_i(d)}$$

其中 k 为平滑系数（项目取 60），$rank_i(d)$ 代表文档 d 在第 i 路召回结果中的排名，n 为召回路径总数。

项目实际采用的是**双层融合策略**（`core/retrieval/fusion.py`），而非纯 RRF 或纯加权求和：

- **第一层：加权合并（`weighted_merge`）**：按照配置的业务权重 `{"text_dense": 0.35, "bm25": 0.20, "graph": 0.25, "image_clip": 0.20}` 对四路召回的原始分数做加权合并，取 Top-K 结果。这一层的意义在于：让业务方可以根据不同查询场景灵活调整各路检索的重要性。

- **第二层：RRF 名次补偿（`reciprocal_rank_fusion`）**：对合并后的结果，叠加 RRF 分数 `item.score += rrf_scores.get(item.item_id, 0.0)`，然后重新排序。RRF 的价值在于：仅依赖排名而非原始分数，天然归一化了不同检索路的量纲差异。

为什么需要双层而不是纯 RRF？纯 RRF 丢弃了原始分数信息，对于分数差异很大的异构检索路（如向量余弦相似度 vs BM25 词频得分 vs 图谱命中计数），完全依赖排名会损失精度。纯加权求和则面临量纲不一致和需要反复调参的问题。双层融合兼顾了两者优势：第一层的业务权重保证了可控性，第二层的 RRF 补偿了排名公平性。

此外，当用户未提供图片输入且文本 Embedding 非 CLIP 模式时，`normalize_weights` 函数会将 image_clip 权重归零并对剩余权重重新归一化，实现了自适应的权重调整。

### 问题4：遇到过某一路召回结果极差拉低整体效果的情况吗？如何处理？

**面试官提问**：遇到过某一路召回结果极差拉低整体效果的情况吗？如何处理？

**回答**：

实际项目中确实存在单路召回质量不稳定的情况。项目设计了多层防护机制：

1. **自动降级**：Neo4j 不可用时，`RetrievalOrchestrator.retrieve()` 中通过 `self.graph_engine.health()` 检测，自动跳过图谱检索路径，返回空列表参与融合。CLIP 不可用时，`EmbeddingProvider.embed_image_paths()` 返回零向量，确保系统不会崩溃。

2. **权重归一化**：无图片输入时，`normalize_weights` 将 image 权重置为 0，剩余三路权重重新归一化，避免无效检索路稀释有效结果。

3. **过召回策略**：各路检索按 `top_k * 2` 或 `top_k * 3` 倍数过召回，保证第一层加权合并时有足够的候选。

4. **调试面板**：开启 `debug=True` 后，`debug_info` 会记录四路检索各自的 Top-5 结果、归一化权重和 RRF 分数，方便定位问题。

针对 BM25 关键词歧义的问题（如"苹果"匹配到水果而非手机），优化方向包括：使用 jieba 分词替代简单空格分词、构建领域词典做实体消歧、设置 BM25 得分阈值过滤低置信度结果。

## 第二模块：文本分块与索引构建

### 问题5：父块+子块的切分策略是如何设计的？

**面试官提问**：采用"父块+子块"的切分策略，如何确定父块和子块的尺寸？有没有参考行业最佳实践？

**回答**：

项目在 `core/ingestion/splitter.py` 中使用 LangChain 的 `RecursiveCharacterTextSplitter` 实现父子分块，具体参数为：

- **父块**：`chunk_size=1500`，`overlap=120`
- **子块**：`chunk_size=380`，`overlap=70`
- **分隔符优先级**：`\n\n` > `\n` > `。` > `.` > ` ` > `""`

父子关联通过 `parent_id` 字段建立。子块从父块内再次切分得到，检索时召回子块（保证精准度），生成时通过 `parent_id` 关联回父块（保证上下文完整性）。

这个参数选择基于以下考量：

- 子块 380 字符约等于 95-100 个 token（项目用 `len(text) // 4` 估算），处于行业推荐的 256-512 token 区间内，保证细粒度检索
- 父块 1500 字符配合 120 的 overlap，保证语义上下文不丢失
- 使用 `RecursiveCharacterTextSplitter` 而非固定长度切分，优先按段落和句子边界切分，减少语义割裂

行业最佳实践通常是子块 256-512 token、父块 2048-4096 token。项目根据自身文档偏短的特性做了适配性调整。

### 问题6：为什么要设计"父块+子块"的切分策略？

**面试官提问**：为什么要设计"父块+子块"的切分策略？普通的固定长度切分有什么问题？

**回答**：

核心目的是解决 RAG 中"检索精度"与"上下文完整性"的两难问题。

普通固定长度切分的问题：不考虑文本语义逻辑，一个完整论点或关联实体可能被割裂到两个块中，导致向量表征不完整、检索时只能召回一半信息、大模型生成时因缺少完整上下文出现幻觉。

项目的设计：

- **子块**（380 字符）：用于 Embedding 向量化，保证检索粒度细，精准匹配用户 Query
- **父块**（1500 字符）：包含子块的完整上下文，检索命中子块后通过 `parent_id` 回溯父块，注入到 Prompt 中
- **上下文构建**：`build_context()` 按顺序拼接检索结果的片段，上限 9000 字符，每段带 `[source:path]` 标记，确保大模型有充足的上下文

同时，项目将父块文本也单独向量化存入 `kb_parent_docs` 集合，构建了子块→父块的双层向量索引，进一步增强了上下文回溯能力。

### 问题7：checksum 增量更新是如何实现的？

**面试官提问**：基于 checksum 实现文档增量识别。如果用户修改了文档，是如何处理旧索引的？

**回答**：

项目使用 SHA-256 算法计算文件校验和，实现在 `core/ingestion/loaders.py` 的 `sha256_file()` 中，按 8192 字节块读取文件计算哈希值。

增量更新流程（`core/indexing/builder.py`）：

1. **加载文档**：扫描 `kb_source/` 目录，加载所有文本和图片文档，计算每个文件的 SHA-256 checksum，同时通过 `uuid5(NAMESPACE_URL, path)` 生成确定性 doc_id
2. **比对 manifest**：`_incremental_filter()` 将当前文档 checksum 与 `IndexManifest` 中已存储的 checksum 比对，仅保留新增或变更的文档
3. **处理变更文档**：对变更文档执行切块、向量化、索引构建
4. **更新 manifest**：将本次处理的文档信息（checksum、source_path、时间戳）写回 `index_manifest.json`

对于修改的文档：新 checksum 与旧 checksum 不同，触发重新切块和索引构建，旧数据通过 Qdrant 的 `upsert` 操作自动覆盖（相同 ID 的点会被更新）。整个流程无需手动清理旧索引。

默认增量构建，`--full` 参数可触发全量重建。

## 第三模块：多模态与 CLIP

### 问题8：多模态 RAG 中如何处理文本、图片的向量嵌入和统一检索？

**面试官提问**：多模态 RAG 中如何处理文本、图片的向量嵌入和统一检索？

**回答**：

项目在 `core/providers/model_provider.py` 的 `EmbeddingProvider` 中统一封装了文本和图像的向量化能力：

- **文本嵌入**：默认使用智谱 embedding-3 API（OpenAI 兼容协议），1024 维，通过 `embed_text()` 批量处理（batch_size=10）。如果配置为 CLIP 模式，则使用本地 SentenceTransformer 编码，与图像共享向量空间。

- **图像嵌入**：使用 CLIP ViT-B-32 多语言模型（本地加载），512 维，通过 `embed_image_paths()` 对图片文件路径编码。CLIP 不可用时返回全零向量作为降级。

- **统一存储**：文本子块存入 `kb_text_chunks` 集合（1024 维），父块存入 `kb_parent_docs` 集合（1024 维），图片存入 `kb_image_assets` 集合（512 维）。通过 Qdrant 的 HNSW 索引实现高效的余弦相似度检索。

- **跨模态检索**：当文本 Embedding 配置为 CLIP 模式时，`RetrievalOrchestrator` 会自动用文本查询向量在 `kb_image_assets` 集合中搜索，实现以文搜图。用户上传图片时，通过 Image Encoder 编码后搜索所有集合，实现以图搜图和以图搜文。

### 问题9：你是如何实现图片触发跨模态检索的？CLIP 在其中扮演什么角色？

**面试官提问**：你是如何实现"图片触发跨模态检索"的？CLIP 模型在这里扮演什么角色？

**回答**：

跨模态检索的核心是 CLIP 模型的图文语义对齐能力。在项目中 CLIP 扮演三个角色：

1. **图像向量化**：知识库构建时，`IndexBuilder.build()` 调用 `embedder.embed_image_paths()` 将所有图片通过 CLIP 的 Image Encoder 编码为 512 维向量，写入 Qdrant 的 `kb_image_assets` 集合。

2. **以文搜图**：当 `EmbeddingProvider` 的文本模式配置为 CLIP 时，`RetrievalOrchestrator.retrieve()` 会用 CLIP Text Encoder 编码用户查询，然后在图片集合中做余弦相似度检索，实现文本搜图片。

3. **以图搜图/搜文**：用户上传图片时，通过 `POST /api/chat/stream` 的 `image` 字段接收，保存到 `runtime/uploads/`，然后通过 CLIP Image Encoder 编码，同时在文本和图片集合中检索。

CLIP 的核心优势：预训练阶段已在海量图文数据上实现了视觉与文本的语义对齐，开源、轻量、部署成本低，通过 SentenceTransformer 封装调用非常简洁。模型缓存统一管理在 `runtime/model_cache/` 目录下。

### 问题10：视觉模型不可用时的降级机制是如何设计的？

**面试官提问**：你特别提到了"视觉模型不可用时的降级机制"。请详细说说你是怎么设计的？

**回答**：

项目在多个层级实现了优雅降级，核心原则是"Fail-Fast & Degradation"——任何单一模块故障不影响系统整体运行：

1. **CLIP 不可用**（`EmbeddingProvider`）：`embed_image_paths()` 中，如果 `_image_model` 为 None（加载失败），返回全零向量 `[[0.0] * 512]`，系统可以继续运行但图片检索效果降级。

2. **Neo4j 不可用**（`LightRAGGraphEngine`）：`health()` 方法通过 `RETURN 1` 测试连接，`RetrievalOrchestrator` 在检索前检查健康状态，不可用时跳过图谱检索，graph_items 为空列表参与融合。

3. **图片输入缺失**：`normalize_weights()` 将 `image_clip` 权重置为 0，其余三路重新归一化（如 Dense 从 0.35 变为 0.4375），保证无图片输入时不会因零分图像结果干扰排序。

4. **Qdrant 批量写入失败**：`QdrantVectorStoreAdapter._upsert_in_batches()` 中，如果写入失败（如超过 32MB payload 限制），自动将 batch_size 减半重试，直到最小为 1 条。

5. **API 健康检查**：`GET /api/health` 接口同时检测 Qdrant 和 Neo4j 的连接状态，前端每 30 秒自动刷新，UI 上显示红/绿状态指示灯。

## 第四模块：SSE 流式输出与 Web 架构

### 问题11：为什么要用 SSE？它和普通的 HTTP 返回有什么区别？

**面试官提问**：为什么要用 SSE？它和普通的 HTTP 返回有什么区别？

**回答**：

项目已经实现了 SSE（Server-Sent Events），在 `webapp/main.py` 的 `chat_stream` 路由中，返回 `text/event-stream` 类型的 `StreamingResponse`。

**普通 HTTP 短连接的问题**：客户端发送请求后需要等待服务端完成所有处理（向量检索 + 结果融合 + 大模型生成完整答案）才能一次性获取结果。大模型生成内容较长时，用户会感到明显的卡顿感。

**SSE 的实现方式**：

```python
async def event_stream():
    yield _sse("meta", {"rewritten_query": rewritten_query, ...})
    for delta in stream:
        yield _sse("token", {"delta": delta})  # 逐 token 推送
    yield _sse("done", {"answer": answer, "retrieval": ...})
```

事件协议定义了四种类型：
- `meta`：元信息（改写后查询、上传图片列表）
- `token`：流式 token 片段，实现"打字机"效果
- `done`：完成事件，包含完整答案和检索结果
- `error`：错误信息

前端通过 `fetch` API + `ReadableStream` 消费 SSE 流（`webapp/static/app.js` 中的 `consumeStream` 函数），使用 `TextDecoder` 解码并按 `\n\n` 分割事件块，逐块解析 `event:` 和 `data:` 行。

选择 SSE 而非 WebSocket 的原因：SSE 部署简单（HTTP 协议无需额外端口）、无需引入 ws 库、与 FastAPI 天然兼容，且本场景是单向推送（服务端→客户端），SSE 完全满足需求。

### 问题12：为什么要从 Streamlit 迁移到 FastAPI？

**面试官提问**：项目从 Streamlit 迁移到 FastAPI + Jinja2，迁移的原因是什么？

**回答**：

V1/V2 使用 Streamlit（`app/streamlit_app.py`），V3 迁移到 FastAPI（`webapp/main.py`），核心原因：

1. **SSE 支持**：Streamlit 的 `st.write_stream` 在流式输出和自定义事件协议上不够灵活，FastAPI 的 `StreamingResponse` 原生支持 SSE，能精确控制事件格式。

2. **前端定制化**：Streamlit 的 UI 组件定制空间有限，迁移到 Jinja2 模板 + 原生 JS/CSS 后，可以实现三栏布局、可折叠侧边栏、自定义调试面板等更精细的 UI 交互。

3. **API 化**：FastAPI 天然提供 RESTful API（`/api/health`、`/api/chat/stream`、`/files`），方便后续扩展移动端或其他客户端接入。

4. **性能**：FastAPI 的异步支持（`async def`）和更轻量的框架开销，在高并发场景下表现更好。

5. **依赖注入**：通过 `bootstrap.py` 的 `@lru_cache` 实现单例 Pipeline 缓存，启动时一次性初始化所有依赖。

旧版 Streamlit 代码保留在 `app/` 目录中作为参考。

## 第五模块：查询改写与多轮对话

### 问题13：多轮对话场景下如何处理上下文记忆？

**面试官提问**：面对多轮对话场景，RAG 系统是如何处理上下文记忆的？

**回答**：

项目已实现多轮对话支持，核心机制是**查询改写**（`core/orchestration/pipeline.py` 中的 `RAGPipeline.rewrite_query()`）：

1. **对话历史传递**：前端通过 `chat_history` 字段将最近 8 轮对话（`chat_history[-8:]`）以 JSON 数组形式传给后端。

2. **查询改写**：使用 `fast_model`（GLM-4-flash，速度快、成本低）将当前问题结合历史对话改写为独立的检索查询。Prompt 为"你是查询优化器，请将用户问题改写为适合检索的单句，不要回答问题本身"，`temperature=0.1` 保证改写结果的稳定性。

3. **上下文注入**：在 `build_answer_messages()` 中，最近 8 轮对话历史同样注入到答案生成的 Prompt 中，保证大模型能延续多轮语境。

4. **SSE meta 事件**：改写后的查询通过 `meta` 事件返回前端，UI 上以 `Rewrite: ...` 标签展示，方便用户了解系统实际检索的内容。

这种方案的优点：轻量级（仅需一次快速模型调用）、无需复杂的对话管理模块、改写失败时 fallback 到原始查询（`return rewritten.strip() or query`）。

## 第六模块：RAG 与微调对比

### 问题14：结合项目说说 RAG 和微调大模型的区别，为什么选择 RAG？

**面试官提问**：结合你的项目，说说 RAG 和微调大模型的区别，你为什么选择 RAG 方案？

**回答**：

| 维度 | RAG（本项目方案） | 微调大模型 |
|------|-----------------|-----------|
| 知识存储 | 外部向量库 + 图谱（Qdrant + Neo4j） | 固化到模型权重中 |
| 知识更新 | 增量更新索引，无需重新训练 | 需重新收集数据、重新训练 |
| 开发成本 | 无需标注数据，代码量可控 | 需大量标注数据和算力 |
| 可解释性 | 检索结果可追溯来源 | 黑盒，难以追溯知识来源 |
| 多模态 | 向量库统一存储文本和图像向量 | 需要多模态微调，复杂度高 |
| 幻觉控制 | 基于检索上下文生成，可检测 | 容易产生过时或编造信息 |

选择 RAG 的核心原因：

1. **适配多模态需求**：项目需要处理文本和图片的统一检索，RAG 通过 Qdrant 统一存储文本和 CLIP 图像向量即可实现，微调方案处理多模态检索的成本极高。

2. **知识更新灵活**：通过 checksum 增量更新机制，添加新文档只需重新索引变更部分，无需重新训练模型。

3. **个人开发场景**：缺乏大规模标注数据和 GPU 算力，RAG 通过 API 调用大模型（智谱AI GLM-4），开发成本低、落地速度快。

4. **可追溯性**：每个检索结果都带 `[source:path]` 来源标记，前端展示文本来源和图片来源，用户可以验证答案的准确性。

5. **已有评估体系**：基于 RAGAS 框架可以系统地评估检索质量和生成质量，微调方案的评估链路更复杂。

## 第七模块：评估体系

### 问题15：你是如何评估检索效果的，使用了哪些评测指标？

**面试官提问**：你在项目中如何评估检索效果，使用了哪些评测指标？

**回答**：

项目集成了完整的基于 RAGAS 0.2.10 的评估系统（`core/evaluation/`），支持两条评估链路：

**线上实时采集**：每次 API 问答后，`EvaluationRecorder` 自动将完整的 QA 数据（query、rewritten_query、检索结果、答案、延迟等）以 JSONL 格式追加写入 `runtime/eval/qa_logs.jsonl`。

**线下批量评估**：使用公开测试集进行标准化评估，支持 5 个数据集（中文：CRUD-RAG、SuperCLUE-C3；英文：HotpotQA、SQuAD v2、Amnesty QA）。评估流程为标准化的 6 步：

1. 加载公开测试集，提取 question/ground_truth/golden_docs/corpus
2. 全量语料库切块、向量化，导入独立的 Qdrant 评估集合（`{dataset}_eval`，与生产知识库隔离）
3. 使用测试问题通过 RAG 系统检索并生成答案
4. 以黄金参考文档为基准计算检索指标
5. 结合标准答案评估生成和端到端指标
6. 输出完整报告与 bad case 分析

**评估指标分三阶段**：

| 阶段 | 指标 | 说明 | 优秀阈值 |
|------|------|------|---------|
| 检索 | `recall@k` | 上下文召回率 | >0.7 |
| 检索 | `precision@k` | 上下文精度 | >0.7 |
| 检索 | `mrr@k` | 平均倒数排名（自定义实现） | >0.7 |
| 检索 | `entity_recall@k` | 实体召回率 | - |
| 生成 | `faithfulness` | 忠实度（幻觉检测，LLM-as-Judge） | >0.8 |
| 生成 | `answer_relevancy` | 答案相关性 | >0.7 |
| 生成 | `answer_coverage` | 答案覆盖率 | - |
| 端到端 | `answer_correctness` | 答案正确性 | >0.7 |
| 端到端 | `answer_similarity` | 答案语义相似度 | >0.7 |
| 端到端 | `coverage@k` | 上下文利用率（自定义） | >0.7 |

其中 MRR 通过检索结果与 golden_docs 前 200 字符的精确匹配实现；answer_coverage 通过答案词集与 ground_truth 词集的交并比实现；coverage@k 通过检索上下文词集与参考文档词集的覆盖率实现。

评估结果自动标注异常样本（低于 0.5 阈值），并给出原因分析（如"答案可能包含幻觉""检索召回率低"等）。报告支持 JSON、CSV 格式导出。

## 第八模块：向量数据库选型

### 问题16：为什么选择 Qdrant 作为向量存储？

**面试官提问**：为什么选择 Qdrant 作为向量存储，没有选择 Milvus 或者 Weaviate？

**回答**：

选型围绕个人独立开发的场景，考量三个维度：

| 维度 | Qdrant（选择） | Chroma（排除） | Milvus/Weaviate（排除） |
|------|---------------|---------------|----------------------|
| 部署复杂度 | 单节点 Docker，API 简洁 | 轻量但功能单一 | 分布式集群，配置复杂 |
| 功能适配性 | HNSW 索引、向量量化、过滤检索、批量写入 | 功能单一，扩展性差 | 功能完善但过度设计 |
| 多模态支持 | 原生支持异构集合（不同维度） | 多模态适配一般 | 支持但资源占用高 |
| 资源占用 | 低，适合单机 | 最低 | 高，需集群环境 |

Qdrant 的具体优势：

1. **批量写入优化**：项目中 `_upsert_in_batches()` 支持自适应分批（默认 64 条/批），超过 32MB 限制自动减半重试，保证大数据量写入的稳定性。

2. **确定性 ID**：通过 `uuid5(NAMESPACE_URL, raw_id)` 将业务 ID 转换为满足 Qdrant 约束的稳定 UUID，支持 `upsert` 幂等操作。

3. **集合管理**：`_ensure_collection()` 按需创建指定维度的集合，余弦距离度量，一行代码搞定。

4. **评估隔离**：评估系统使用独立的 `{dataset}_eval` 集合，不影响生产知识库，评估完可一键清理。

## 第九模块：幻觉控制与提示词设计

### 问题17：大模型生成阶段出现幻觉问题，有什么优化手段？

**面试官提问**：大模型生成阶段出现幻觉问题，你采取了什么优化手段？

**回答**：

项目从多个维度控制幻觉：

1. **Prompt 约束**（`core/generation/prompting.py`）：System Prompt 明确要求"必须优先依据给定上下文回答；若证据不足明确说不知道；给出简洁回答并附来源"，从源头限制幻觉。

2. **上下文完整性**：通过父子分块策略，生成时注入父块内容（而非仅子块），确保大模型有完整的上下文信息，减少因信息不完整导致的幻觉。

3. **检索质量保障**：四路召回 + 双层融合 + 过召回策略，尽量减少不相关的检索结果输入大模型。

4. **faithfulness 指标监控**：评估系统中使用 RAGAS 的 `faithfulness` 指标（LLM-as-a-Judge）检测幻觉，自动标注低 faithfulness 的样本为异常 case。

5. **来源标注**：生成答案时每个上下文片段都带 `[source:path]` 来源标记，Prompt 要求"附来源"，用户可以追溯验证。

6. **temperature 控制**：答案生成使用 `temperature=0.3`（查询改写为 0.1），降低随机性，提高答案的确定性和一致性。

## 第十模块：扩展性思考

### 问题18：如果需要处理百万级海量数据，系统架构需要做哪些改造？

**面试官提问**：如果需要处理百万级海量数据，你的系统架构需要做哪些改造？

**回答**：

当前项目面向个人知识库场景（文档量在千级别），若扩展到百万级，需从四个维度改造：

1. **存储层**：Qdrant 单机替换为分布式集群，实现数据分片和负载均衡；或升级为 Milvus 分布式向量库。引入 Redis 缓存高频查询结果。

2. **检索层**：当前四路串行检索改为并行异步，使用 asyncio 并发调用 Dense、BM25、Graph、Image 四路检索。引入向量量化（Product Quantization）降低计算成本。

3. **数据摄入层**：引入分布式文档处理框架（如 Spark），大规模切块和嵌入并行化。当前批量嵌入 batch_size=10 需要显著提升。

4. **服务层**：从单进程 FastAPI 改为多 Worker 部署（`uvicorn --workers N`），或拆分为微服务架构，检索、生成、评估模块独立部署和扩展。

5. **评估层**：当前评估使用同步 RAGAS 调用，百万级数据需要异步批处理，引入消息队列（如 Celery + Redis）管理评估任务。

### 问题19：针对领域专属词汇，向量模型无法很好理解，你是怎么解决的？

**面试官提问**：针对领域专属词汇，向量模型无法很好理解，你是怎么解决的？

**回答**：

当前项目使用智谱 embedding-3 API 作为文本嵌入模型，已经具备较好的中文理解能力。针对领域专属词汇，有以下优化方向：

1. **jieba 自定义词典**：项目中已引入 jieba 用于图谱实体提取，可扩展为自定义词典，添加领域术语（如 `jieba.add_word("CLIP", freq=1000, tag='nz')`），确保 BM25 分词时能正确切分领域词汇。

2. **查询预处理**：在查询改写 Prompt 中增加领域词汇标准化指令，让 fast_model 自动将缩写和别名映射为标准术语。

3. **实体消歧**：结合图谱中的实体关系，在查询改写阶段自动补全实体上下文（如"苹果"→"苹果公司"）。

4. **Embedding 模型微调**：使用领域内文本数据对 CLIP 或其他 Embedding 模型做轻量化微调，让向量空间更好地编码领域语义。

## 评估体系设计哲学

### 核心问题：怎么判断 RAG 系统好不好？

一个 RAG 系统本质做两件事：**检索**和**生成**。所以评估围绕这两件事展开。

### 检索评估

给系统一个问题，看检索结果和"应该找到的文档"差距多大。

| 角度 | 指标 | 通俗解释 |
|------|------|----------|
| 找没找到 | `recall@k` | golden_docs 里的关键信息，检索结果里有没有 |
| 准不准 | `precision@k` | 检索回来的文档，有多少是真正相关的 |
| 排得对不对 | `mrr@k` | 第一个相关文档排在第几位 |
| 覆盖够不够 | `entity_recall@k` | 检索上下文里的实体，答案有没有提到 |

### 生成评估

无标准答案时（LLM-as-a-Judge）：

| 指标 | 通俗解释 |
|------|----------|
| `faithfulness` | 答案有没有"编"东西 |
| `answer_relevancy` | 答案有没有答非所问 |

有标准答案时：

| 指标 | 通俗解释 |
|------|----------|
| `answer_correctness` | 答案对不对 |
| `answer_similarity` | 答案和标准答案有多像 |

### 设计哲学

1. **检索和生成分开评**：检索不好，生成再好也没用；生成不好，检索再准也白搭。
2. **有标准答案评正确性，没标准答案评合理性**：`faithfulness` 不依赖标准答案——"你找来的文档里没有说过这句话，就是幻觉"。
3. **评估环境与生产环境隔离**：评估数据导入独立的 `{dataset}_eval` 集合，不影响线上用户的检索体验。评估完可以一键清理。
